# BuildX Architecture

Turn a one-line software requirement into a full engineering plan — PRD, database schema, REST API spec, React component tree, sprint backlog, and Docker setup — generated by a chain of small, specialized AI agents instead of one giant prompt.

> Input: `"Build a CRM for a logistics company"`
> Output: PRD → Schema → API → Component Tree → Sprint Plan → Docker files

## Why this exists

Most "AI generates your app" tools either hide everything behind a black box or hand back unreviewable code. BuildX Architecture does the opposite: it produces the **planning artifacts** a human engineering team would produce before writing code, with a review/regenerate checkpoint after every stage. The bet is that AI + human-in-the-loop beats full automation for this kind of work — each stage's output can be inspected and re-run independently before the next stage builds on it.

## How it works

```
spec ──▶ Requirements Agent ──▶ Schema Agent ──▶ API Agent ──▶ Frontend Agent ──▶ Sprint Agent
                  │                  │               │               │                │
                  ▼                  ▼               ▼               ▼                ▼
                PRD              DB Schema       REST API      Component Tree     Sprint Plan
```

- Each stage is a **separate Gemini call** with its own Pydantic schema enforced via `response_schema` — guaranteed structured JSON, no manual parsing.
- Stages are sequential: each agent's prompt is built from the **previous stage's output**, not the raw spec.
- Docker files (`Dockerfile` × 2 + `docker-compose.yml`) are **templated, not LLM-generated** — more reliable and costs zero tokens.
- `/generate` **streams** results stage-by-stage over newline-delimited JSON, so the UI lights up each tab as soon as that stage finishes instead of blocking on the full ~20–30s pipeline.
- Every stage has a **Regenerate** button that re-runs just that one agent with the same upstream context — catches and fixes a bad output (e.g. a hallucinated schema relationship) without restarting the whole plan.
- Token usage and estimated cost are tracked **per stage and cumulatively**, surfaced live in the UI footer.

## Tech stack

**Backend** — FastAPI, Gemini 2.5 Flash (`google-genai`), Pydantic, slowapi (rate limiting)
**Frontend** — React 18, Vite, Tailwind CSS

## Project structure

```
backend/
├── app/
│   ├── main.py              # FastAPI app, /generate (SSE) and /regenerate endpoints, rate limits
│   ├── schemas.py           # Pydantic models for every stage's output
│   ├── gemini_client.py     # Gemini call wrapper — structured output + token usage
│   ├── agents.py            # One function per agent (requirements/schema/api/frontend/sprint)
│   ├── cost.py               # Token → estimated USD cost conversion
│   └── docker_templates.py  # Templated Dockerfile / docker-compose generation
├── requirements.txt
└── .env                      # GEMINI_API_KEY=...

frontend/
├── src/
│   ├── App.jsx                       # Streams /generate, tracks stage status + running cost
│   ├── components/
│   │   ├── SpecInput.jsx             # Input box + Generate button
│   │   ├── ResultTabs.jsx            # Tab bar — disabled until each stage's data arrives
│   │   ├── RegenerateButton.jsx      # Per-stage re-run, reports token/cost back up
│   │   ├── StageProgress.jsx         # Pending → in-progress → done chip row
│   │   ├── CostFooter.jsx            # Running token count + estimated cost
│   │   └── tabs/
│   │       ├── PRDTab.jsx
│   │       ├── SchemaTab.jsx         # Tables + relationships
│   │       ├── ApiTab.jsx            # Endpoints grouped by method
│   │       ├── ComponentsTab.jsx     # Rebuilds tree from flat parent_id list
│   │       ├── SprintTab.jsx         # Sprint columns with story points
│   │       └── DockerTab.jsx         # Copyable code blocks, no regenerate (templated)

│       └── api.js                    # Streaming fetch reader + 429 handling
├── package.json
├── vite.config.js
├── tailwind.config.js
└── .env                               # VITE_BACKEND_URL=http://localhost:8000
```

## Setup

### Backend
```bash
cd backend
pip install -r requirements.txt
```
Create `backend/.env`:
```
GEMINI_API_KEY=your_key_here
```
Run:
```bash
uvicorn app.main:app --reload --port 8000
```

### Frontend
```bash
cd frontend
npm install
```
Create `frontend/.env`:
```
VITE_BACKEND_URL=http://localhost:8000
```
Run:
```bash
npm run dev
```
Opens on `http://localhost:5173`.

## API

### `POST /generate`
Streams the full pipeline as newline-delimited JSON. Rate limited to **5/hour per IP**.

Request:
```json
{ "spec": "Build a CRM for a logistics company" }
```

Each line of the response body is one JSON object:
```json
{"stage": "requirements", "status": "start"}
{"stage": "requirements", "status": "done", "data": {...}, "usage": {...}, "running_tokens": 412, "running_cost": 0.000183}
{"stage": "schema", "status": "start"}
...
{"stage": "complete", "total_tokens": 2104, "total_cost": 0.000940}
```

### `POST /regenerate`
Re-runs a single stage with its required upstream context. Rate limited to **20/hour per IP**.

Request:
```json
{
  "stage": "schema",
  "requirements": { "...": "previous requirements output" }
}
```

| stage | requires |
|---|---|
| `requirements` | `spec` |
| `schema` | `requirements` |
| `api` | `requirements`, `schema_design` |
| `frontend` | `requirements`, `api` |
| `sprint` | `requirements`, `schema_design`, `api`, `frontend` |

Response:
```json
{ "data": {...}, "usage": { "input_tokens": 120, "output_tokens": 340, "total_tokens": 460 }, "cost": 0.000201 }
```

### `GET /status`
Health check (not `/health` — ad blockers commonly block that path).

## Rate limiting

In-memory, per-IP, via `slowapi`. `/generate` is capped tighter than `/regenerate` since it runs 5 LLM calls per request vs. 1. Exceeding the limit returns `429` with `{"detail": "Rate limit exceeded..."}`, surfaced directly in the UI's error banner.

## Cost tracking

Token counts come straight from Gemini's `usage_metadata` on every response — not estimated. Cost is computed from a hardcoded per-model rate table in `cost.py` (placeholder figures — check [ai.google.dev/pricing](https://ai.google.dev/pricing) before treating these as real billing numbers). This is also the natural integration point with a usage-ledger style cost-observability tool, since every agent call already reports `{input_tokens, output_tokens}`.

## Known limitations

- Docker output is templated from the schema/API shape, not validated against the actual stack choices in the PRD.
- No persistence — refreshing the page loses the current plan. Stateless by design for v1.
- Schema/relationship direction is not cross-validated between stages (e.g. a relationship can point the wrong way relative to its actual foreign key) — this is exactly what the per-stage Regenerate button exists to catch.
